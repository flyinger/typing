# Practice UI Review

TypingLab 的训练台不是测速玩具，也不是课程站首页。它应该像一个长期使用的工作台：打开后马上练，输入时只给必要反馈，结束后给诊断和下一步。

## 参考界面

### Typing.io

可借鉴：

- 用真实代码作为材料，保留符号、缩进、大小写和多行结构。
- 训练区域应该让代码本身成为主视觉，而不是把代码藏在小卡片里。
- 对代码训练来说，行列位置、下一字符和错误位置比营销式统计更重要。

不照搬：

- Typing.io 更偏代码专项；TypingLab 还要覆盖五笔、中文真实输入、英文和 Vim。
- 代码编辑器不能替代中文 IME 的 textarea；中文真实输入仍要优先保证 composition 事件可靠。

### edclub / TypingClub

可借鉴：

- 课程化进度清楚，用户知道当前练的是哪一课、为什么练。
- 弱键、准确率和进度要在训练后转化为下一课或复练建议。
- 输入中反馈要稳定，不应因为数字跳动破坏节奏。

不照搬：

- 它的游戏化和学校课堂元素不适合 TypingLab 的工作流训练。
- 不需要大面积动画、徽章和分数系统。

### 10FastFingers

可借鉴：

- 当前词高亮是高效设计：用户知道眼睛该落在哪里。
- 主要反馈是当前输入、错误、倒计时和速度，界面很克制。
- 词流适合英文底座，但需要扩展到代码、中文和 Vim 序列。

不照搬：

- 随机词列表不能支撑真实工作输入。
- 只测速度会弱化退格、停顿和提示依赖这些长期改进指标。

### Monkeytype / keybr

可借鉴：

- Monkeytype 的实时反馈很轻，复杂统计放在结束后。
- keybr 的弱键/弱项复练适合 TypingLab 的自适应队列。
- 训练中应保留键盘优先操作，鼠标只做辅助。

不照搬：

- TypingLab 不应追求过度自定义外观，长期效率比主题玩法重要。
- 弱键算法不能只看英文键位，还要兼容五笔字根、中文候选、代码符号和 Vim motion。

## 当前设计原则

1. 训练中只显示五类信息：目标、下一键、当前片段、实时质量、完成动作。
2. 复杂统计放在完成后的诊断和 Analytics，不抢输入时注意力。
3. 短句用于第一阶段建立稳定样本；长段落和多行代码用于第二阶段真实节奏。
4. 中文真实输入优先 textarea，代码训练优先 Monaco。
5. 完全匹配自动保存，`Enter` 进入下一步；处方要求重练时，主按钮和 Enter 行为一致。
6. 实时速度前几秒显示为预热，避免短样本 CPM 虚高。

## 本轮落地

- 目标区改成逐字状态流：正确、错误、当前、未输入和额外输入分色。
- 目标区增加操作仪表：下一键、当前词/片段、行列、剩余、错误。
- 代码目标使用等宽显示，保留多行和缩进。
- 实时状态条继续显示 CPM、准确率、进度、退格/100字，但输入前几秒对速度做预热提示。
- 内置材料增加英文连续段落、多行 TypeScript 和连续命令片段。

## 后续优化

P0：

- 当前词/当前片段可点击展开上下文，方便长段落里重新定位。
- 代码模式在 Monaco 内高亮目标差异，而不是只在上方目标流中高亮。
- 中文真实输入增加“候选压力”提示：composition 次数、退格和长停顿结合判断。

P1：

- 按模式切换目标流密度：英文显示词流，代码显示行列，五笔显示字根/编码卡片，中文显示句段。
- 增加错误回放：完成后高亮最早错误、重复错误和最长停顿位置。
- 增加移动端训练布局，保证目标流和输入区不互相挤压。

P2：

- 课程化 lesson UI：把第一阶段底座、第二阶段多行、第三阶段中文迁移做成可连续完成的课程轨道。
- 可选键盘热力浮层，但默认不显示，避免干扰。
