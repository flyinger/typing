# TypingLab

TypingLab 是一个本地优先的个人打字训练系统，面向从拼音迁移到 Rime 五笔 86 的长期训练，同时覆盖中文真实写作、英文/术语、代码和 Vim 操作序列。

## 当前实现范围

- React + Vite + TypeScript Web MVP
- 五笔编码训练：直接输入编码，支持 `z` / `?` 提示和多编码校验
- 真实中文训练：textarea 捕获 `keydown`、`input`、`compositionstart/update/end`
- 代码训练：Monaco Editor 输入区
- 本地 IndexedDB 缓存：sessions、events、materials、settings
- append-only 事件模型：`deviceId + sessionId + sequence` 生成稳定 `eventId`
- 手动同步包导入/导出：跨 Mac / Ubuntu 合并训练结果
- 同步与数据健康面板：显示设备分布、事件类型、完整性问题和下一步同步建议
- 同步行动队列：按“现在、下一步、之后”提示导出、等待文件同步、另一台导入、合并后再导出的跨设备闭环
- 同步目录文件清单导入/导出：生成并读取 `TypingLab/manifest.json`、`sessions/*.jsonl`、`materials/*.json`、`snapshots/*.json`、`exports/*.csv` 的落盘计划
- 同步目录直接读写：支持 File System Access 的浏览器可把同一套 `TypingLab/` 文件结构写入用户选择的本地目录，也可从该目录读回并预览合并
- 事实流摘要：同步包、同步目录 manifest、snapshot 和 Settings 健康面板都会显示同一套 `sessions/events/materials` 摘要，用于确认 Mac / Ubuntu 是否已合并到同一状态
- 材料合并预览：导入前显示同名同内容跳过、同名不同内容保留版本、同 ID 不同内容改名导入和会话重映射
- Rime 五笔词库导入：支持 `wubi86_jidian.dict.yaml` 一类文件
- Materials 底座材料准备度：提前检查英文/代码材料是否足够长、足够多，避免短材料无法计入 80 CPM 稳定窗口
- 专业材料生成器：从项目笔记、技术方案、命令记录或代码片段中拆出中文、英文、代码和 Vim/命令训练材料
- CSV、JSON、JSONL 导出
- 内置五笔教程和样例材料
- 内置英文/代码/Vim 速度底座材料：可直接开始第一阶段训练
- Analytics 底座冲刺视图：集中查看英文/代码样本、速度、准确率和退格门槛
- Analytics 周复盘视图：自动生成本周结论、风险、模式投入、下周训练计划和验收标准；计划块可直接启动对应训练队列
- 周复盘 Markdown 导出：可把本周决策、成果、风险、弱项和下周计划保存为本地 Markdown 文件
- 周复盘主线焦点：总体均值只作背景，速度变化、风险和升级判断优先看当前执行协议的主线模式，避免英文/代码/五笔/中文混合均值误导训练决策
- Analytics 训练决策视图：训练日历热力、键位错误热力、五笔根键压力、错误位置榜和长停顿位置榜
- Today 执行协议：根据当前阶段显示升级标准、保护规则和复盘动作
- Today 速度底座冲刺方案：把“英文/代码先到 80 CPM，再用 100 CPM 做维护舒适线”的策略转成可执行训练块、解锁预估和每日规则
- Today 协议感知队列：英文/代码底座未过时优先推英文/代码，五笔只作为维护；底座达标后再把五笔/真实中文推到主位，点击 `开始今日训练` 后按队列连续推进
- Today 队列可执行性：开始前显示今日计划可展开几组、已有材料能覆盖几组、缺哪些模式，并联动英文/代码底座材料准备度
- Practice 底座有效性：英文/代码训练中实时显示本轮是否会计入 5/20 轮底座窗口，差多少秒/输入单位，以及提示、粘贴是否已使本轮变成热身
- Practice 实时反馈：输入中显示 CPM、准确率、进度和退格/100字；目标完全匹配会自动保存，`Enter` 可完成当前题或进入下一组
- Practice 目标流高亮：目标文本会逐字显示已正确、当前字符、错误字符、未输入字符和额外输入，并显示下一键、当前词/片段、行列、剩余和错误；代码目标保留换行和等宽排版。界面评估见 [docs/practice-ui-review.md](docs/practice-ui-review.md)
- Practice 手动模式保护：自由练习切到非当前主线时会提示它只是维护/加练，避免底座未达标时误把五笔当主训练
- Practice 未完成输入保护：训练中已有输入但未完成时，退出、下一组、重练、切模式或关闭页面都会先确认
- 训练队列覆盖提示：Today 和周计划会按计划分钟估算应练组数，材料不足时提示缺几组，避免计划静默缩短
- Today 训练路线图：把英文/代码底座、五笔编码准确、真实中文迁移、生产切换四阶段显示为可计算状态；80 CPM 是五笔主线解锁线，100 CPM 是英文/代码维护舒适线
- Tutorial 动态训练日程：按当前数据生成未来 20 个训练日，明确底座推进、80 CPM 切换复测、五笔主线、每 5 天复盘和同步节点
- 本地启动器：`npm run open` 启动可关闭的本地预览服务，`npm run stop` 清理服务，`npm run install:launchers` 生成 macOS/Ubuntu 桌面入口
- 自适应训练队列：按错误、提示、退格、停顿、复习间隔生成下一组建议
- 12 周训练路线：先把英文/代码输入推到 80-100 CPM，再进入五笔、真实中文和生产切换
- 生产构建包含 PWA manifest 和基础离线缓存

## 启动

推荐日常训练入口：

```bash
npm run open
```

`npm run open` 会通过 `bash scripts/open-typinglab.sh` 使用生产预览模式启动本地页面，不依赖脚本文件的 executable bit，默认从 `127.0.0.1:5173` 开始找可用端口。启动器打开页面时会带一次性 `typinglab_launch` 参数，降低浏览器或 PWA 缓存继续显示旧页面的概率。如果新电脑刚 clone 下来还没有 `node_modules/`，启动器会先自动安装依赖：有 `package-lock.json` 时执行 `npm ci`，否则执行 `npm install`，安装完成后继续打开训练页。如果服务已经在当前工作区运行，它会直接打开已有地址而不是重复启动。关闭训练时可以直接关闭这个终端，或运行：

```bash
npm run stop
```

启动器会检查 Node.js / npm，并要求 Node.js 18+。macOS 桌面入口会主动补上常见 Homebrew 路径 `/opt/homebrew/bin` 和 `/usr/local/bin`；如果你用 nvm/asdf 且双击入口找不到 Node，可以在终端里运行 `npm run open`。

生产预览会在 `dist/index.html` 不存在，或 `src/`、`public/`、`index.html`、`package.json`、`package-lock.json`、`tsconfig.json`、`vite.config.ts` 比当前构建更新时自动执行 `npm run build`，避免源码更新后仍打开旧版本。

如果公司网络限制 npm，或你想手动控制依赖安装，可以先运行 `npm ci`；也可以用 `TYPINGLAB_AUTO_INSTALL=0 npm run open` 关闭自动安装，此时缺依赖会只提示错误。

如果想生成更接近产品形态的桌面入口：

```bash
npm run install:launchers
```

这会先修复本地启动脚本权限，然后生成：

- `launchers/TypingLab.command`：macOS 可双击启动
- `launchers/typinglab.desktop`：Ubuntu 可复制到 `~/.local/share/applications/`

这些入口本质上仍是本地 Web MVP：打开时启动本地服务，关闭终端或运行 `npm run stop` 后停止，不保留后台常驻进程。最终桌面产品会用 Tauri 打包成 macOS `.app` 和 Ubuntu AppImage/deb。

开发调试仍可使用：

```bash
npm run train
```

`npm run train` 会启动 Vite 开发服务并自动打开浏览器。默认地址是 `http://127.0.0.1:5173/`。

如果只想启动服务、不自动打开浏览器：

```bash
npm run serve
```

如果想用启动器进入开发模式，而不是生产预览模式：

```bash
TYPINGLAB_MODE=dev npm run open
```

也可以直接运行：

```bash
./scripts/start-typinglab.sh
```

如果 macOS 上 `npm install` 卡在 `fsevents` 可选依赖构建，可以改用：

```bash
npm install --omit=optional
npm install --include=optional --ignore-scripts
```

第二步会补齐 Rollup/Vite 需要的平台包，同时跳过卡住的可选构建脚本。

## 跨设备同步模型

MVP 不维护远端数据库。权威数据是可追加、可去重、可合并的训练事实流：

```text
TypingLab/
  sessions/*.jsonl
  materials/*.json
  snapshots/*.json
  exports/*.csv
```

公开 GitHub 仓库默认只用于分发 TypingLab 代码、内置材料和启动脚本，不默认保存个人训练记录。`TypingLab/` 同步目录、`typinglab-sync-*.json`、`typinglab-sync-folder-*.json`、`typinglab-events-*.jsonl`、`typinglab-sessions-*.csv` 和 `typinglab-weekly-review-*.md` 已加入 `.gitignore`，避免把个人输入内容、错题、停顿、周复盘和设备信息误提交到公开仓库。`npm run doctor` 还会扫描项目目录内是否残留这些同步产物；如果确实临时把同步目录放在项目目录里，可用 `TYPINGLAB_ALLOW_LOCAL_SYNC_DATA=1 npm run doctor` 显式放行，但发布公开仓库前应移走。训练记录跨 Mac / Ubuntu 合并仍优先使用同步目录、同步包、Syncthing、网盘或公司允许的文件传输；如果要把记录放进 GitHub，优先另建私有数据仓，公开仓只放脱敏汇总。具体评估和验收方案见 [docs/github-sync.md](docs/github-sync.md)。

当前 Web MVP 先提供四种同步路径：

1. 在 Mac 上训练后进入 `Settings`，导出同步包。
2. 如果浏览器支持 File System Access，在 `Settings` 选择 `写入同步目录`，把 `TypingLab/` 直接写到 Syncthing、网盘或公司允许的同步文件夹。
3. 在另一台电脑同步完成后，选择 `读取同步目录`，从同步父目录或 `TypingLab` 目录读回事实流，先查看新增/跳过预览，再确认合并。
4. 如果浏览器不支持本地目录读写，导出 `同步目录清单`，在另一台设备导入清单预览后确认合并。
5. 反向也一样。重复导入会按 `eventId` 和 `sessionId` 去重。

导入时会先校验同步包结构；同名同内容材料会跳过，同名不同内容材料会保留两个版本，同 ID 但内容不同的材料会生成新 ID，并把导入会话的 `materialId` 重映射到新材料。Settings 的待确认导入面板会在确认前列出材料合并说明，避免跨设备材料冲突静默发生。

`Settings` 会记录上次同步包导入/导出时间，并在“同步与数据健康”里显示设备分布、事件类型、材料引用问题和结构化同步行动队列，方便你在 Mac 和 Ubuntu 间判断是该导出、等待文件同步、导入另一台、重新导出合并结果，还是先修数据问题。

同步健康会额外显示 `待导出会话`、`待导出事件` 和 `待导出材料`：只要上次导出后又完成训练、产生事件、导入 Rime 词库、创建专业材料或恢复内置材料，Settings 就会提示需要重新导出同步包或写入同步目录。若最近一次导入晚于最近一次导出，也会提示重新导出，确保本机合并后的完整数据可以被另一台电脑继续合并和备份。

Today 首屏和 Practice 完成后的诊断区也会显示待导出提醒，并提供 `打开同步设置` 入口。训练中不弹同步提示，训练结束或回到 Today 时再处理同步，避免打断输入节奏。

`写入同步目录` 会请求浏览器目录权限。如果你选择的是同步根目录，应用会在下面创建 `TypingLab/`；如果你直接选择 `TypingLab` 目录，应用不会再创建嵌套的 `TypingLab/TypingLab`。写入成功后，本次打开应用期间会保留目录授权，后续训练完成、导入合并、材料导入、新建材料、删除未使用材料或恢复内置材料后，都会尝试自动写入同一同步目录；如果浏览器要求重新授权，Settings 会提示再次手动点击 `写入同步目录`。写入时会先落盘 sessions、materials、snapshots 和 exports，最后写入 `manifest.json`，降低另一台设备读到半份新同步结果的概率。`读取同步目录` 支持同样的两种选择方式，并复用同步包的待确认预览与去重合并规则。该能力依赖浏览器支持，受限环境仍可用同步包或同步目录清单。

`同步目录清单` 会导出一个 JSON，里面包含桌面版需要写入同步目录的文件路径、媒体类型和内容。清单可以导入另一台设备并按 `eventId` / `sessionId` 去重合并。Tauri 版会复用同一套清单和写入逻辑把文件落到 `TypingLab/`。

读取同步目录或同步目录清单时，会先按 `TypingLab/manifest.json` 校验文件是否齐全、字节数是否一致。若 Syncthing、网盘或手动拷贝还没同步完成，应用会拒绝导入并提示等待完成，避免把半份事实流合并进本机 IndexedDB。若 Mac 和 Ubuntu 在互相导入前分别写入了同一个同步目录，最后写入的 manifest 可能只列出其中一台设备已知的文件；应用会先校验 manifest 已列文件，再聚合目录里其它完整的 `sessions/*.jsonl` 和 `materials/*.json`，用于形成可合并的多设备事实流。

同步包和同步目录还会携带事实流摘要。这个摘要按 `sessions`、`events` 和 `materials` 的稳定内容计算，不受导入顺序影响；manifest 已列文件如果字节长度看起来完整但内容与摘要不一致，应用会拒绝导入。目录中额外的完整设备文件会进入聚合摘要。同步包和同步目录清单的文件名会包含精确 UTC 时间戳和 12 位摘要，例如 `typinglab-sync-20260629T070809Z-1234567890ab.json`，同一天多次导出也能区分。完成双向同步后，Mac 和 Ubuntu 的 `Settings -> 同步与数据健康 -> 本机摘要` 应该一致；这可以作为跨设备训练记录已经合并的快速验收信号。

桌面版计划使用 Tauri，把事件日志自动写入同步目录，保留同一套合并规则。

## 导入 Rime 五笔词库

进入 `Materials`，选择 `导入 Rime 五笔词库`，导入例如：

```text
wubi86_jidian.dict.yaml
wubi86_jidian_user.dict.yaml
```

解析规则：

- 跳过 YAML header
- 从 `...` 后读取词条
- 识别 `词条<TAB>编码<TAB>权重`
- 编码统一转为小写，只接受 1-8 位英文字母编码，跳过空编码、数字编码和异常行
- 同一词条多个编码会合并
- 如果词库正文没有任何有效词条，导入会失败并在页面提示原因，不会创建空材料包

## 导入自定义材料

内置启动材料已经包含五笔一级简码、25 个字根键位 drill、真实中文段落、英文速度底座、代码符号、命令行和 Vim/命令 kata，足够开始第一阶段训练。真实中文材料覆盖日常计划、同步合并、机器人/仿真/视觉伺服、技术复盘和周复盘语境；英文/代码材料覆盖 Git、CLI、TypeScript、Python、jq/curl、隐私同步和事件日志语境。短句和单行代码主要用于第一阶段建立 80 CPM 底座样本，不是最终形态；内置库还包含第二阶段的英文连续段落、多行 TypeScript 和连续命令片段，用于后续过渡到真实工作节奏。后续可以继续导入自己的专业词、项目报告、命令和代码片段。

材料长度按阶段推进：

- 第 1 阶段：短英文句、单行代码、命令和 Vim kata，用来控制变量，稳定测出速度、准确率和退格。
- 第 2 阶段：长英文段落、多行代码和连续命令，开始训练换行、缩进、标点和真实节奏。
- 第 3 阶段：真实中文段落、技术笔记、项目报告和专业词，重点观察候选、退格和停顿。
- 第 4 阶段：工作流材料来自真实项目笔记、终端历史、代码 diff 和周复盘。

`Materials` 里的文本导入支持每行一条。普通中文、英文、代码和 Vim 材料会把整行作为目标文本。五笔编码材料必须同时包含目标字词和至少一个有效英文字母编码；只有目标字词、只有数字编码或混入非字母编码的行会被跳过，如果整份材料没有有效行，导入会失败并提示原因。

`Materials` 顶部会显示英文/代码底座材料准备度。80 CPM 解锁线要求单轮至少 12 秒，因此材料至少需要约 16 个输入单位才适合计入稳定窗口；100 CPM 舒适线建议至少 20 个输入单位。每个模式至少准备 10 条这样的材料，才能支撑每日计划和周计划不反复用同一两条短材料。

如果材料还没有保存成文件，可以在 `新建材料包` 里直接粘贴内容，选择训练模式后创建。适合临时加入专业词、项目报告句子、代码片段或 Vim kata。也可以先在 `专业材料生成` 粘贴项目笔记、技术方案、命令记录或代码片段，应用会本地拆分出中文真实输入、英文术语、代码片段和 Vim/命令候选；生成结果可以填入编辑后创建，也可以直接创建材料包。

五笔编码材料支持附带编码：

```text
器械	kkaw kkag
视觉伺服,pywx/wwfy
```

多个编码可以用空格、`/` 或 `|` 分隔。

材料包删除规则：

- 内置材料不能删除，可以用 `恢复内置材料` 覆盖恢复。
- 已被训练记录引用的材料不能删除，避免同步包里出现悬空材料引用。
- 未使用的导入材料可以删除，适合清理误导入的词库或文本。

## 训练入口

- `Today`：打开即练，显示今日目标、连续天数、弱项和趋势
- `Practice`：五笔、中文、英文、代码、Vim 五类训练
- `Analytics`：长期趋势、英文/代码底座冲刺、周复盘、模式分布、弱字/弱词、弱键位
- `Materials`：导入 Rime 或文本材料，搜索/过滤材料包，查看样例，删除未使用的导入材料，恢复内置材料
- `Tutorial`：12 周五笔路线、拆字/识别码/词组/个人词库教程和练习建议
- `Settings`：设备 ID、每日目标、同步健康、同步目录提示、同步包、同步目录清单、JSONL、CSV、本机数据维护

`Today` 会根据今日剩余分钟生成训练安排，并显示当前阶段目标、训练路线图和今日执行协议。剩余时间少于 15 分钟时，安排会收敛成一个主线聚焦块；今日目标完成后不再显示必练清单，只保留自由加练入口。当前整体策略是：先把英文/代码输入速度提升到 80 CPM 解锁线、准确率稳定到 96% 以上，再把主要训练时间切给中文五笔；英文/代码继续用每日维护量推向 100 CPM 舒适线。这样做的原因是英文、命令、代码符号和 Vim 动作是工作流底层输入能力，先打顺可以降低日常开发摩擦，也能让后续五笔训练更稳定。

应用打开后会按本地分钟刷新 Today 状态；跨午夜时今日分钟数、连续天数、30 日趋势、同步健康和 14 天复测有效期会自动更新，不需要重新打开页面。

`Today` 的 `速度底座冲刺方案` 会把这条策略拆成当前焦点、预计解锁轮次、最快/保守训练日范围、今日训练块、执行规则和里程碑。英文和代码各完成 5 轮有效样本只算基线；各累计到 20 轮有效稳定窗口，最近 14 天内有复测，并且近 20 轮有效样本平均达到 80 CPM、准确率和退格达标后，才把主训练切到五笔和真实中文。有效样本要求单轮至少 12 秒，材料和实际输入都至少 10 个输入单位，且没有提示、粘贴或大段突增输入事件；太短的材料、未完成输入、热身轮次、误粘贴、提示依赖或自动填充不会用于解锁五笔主线。100 CPM 不阻塞中文五笔，只作为英文/代码长期维护目标。

`Tutorial` 的 `未来 20 个训练日` 会把长期路线变成具体日程：底座未达标或复测过期时先安排英文/代码推进日和第 5 天复测；顶部会显示按今日队列容量估算的最快训练日和按质量折损估算的保守训练日。日程切换仍按保守估算展示，避免还没产生合格样本就提前承诺五笔主线；预计达标进入视窗后再安排 80 CPM 解锁复测，随后切到五笔字根、弱字弱词、真实中文和专业词。每 5 个训练日都要求复盘并写入同步目录或导出同步包。

切换到五笔主线前，`Today` 会显示 12 个底座门槛：英文/代码各 5 轮有效基线样本、英文/代码各 20 轮有效稳定窗口、英文/代码最近 14 天内复测、英文/代码近 20 轮有效样本速度都 >= 80 CPM、准确率都 >= 96%、英文退格 <= 10/100 字、代码退格 <= 12/100 字。门槛判断会按 `startedAt` 取最近有效样本，不依赖 IndexedDB 或同步包导入顺序；旧高分不会掩盖最近退步，长期停练后的旧高分也不会让五笔主线继续假解锁。底座过 80 CPM 后路线图会标记为“已解锁”，但只有英文和代码都到 100 CPM 才标记为“已达成”，每日维护块会自动选择当前较弱的英文或代码。

`Analytics` 的 `底座冲刺` tab 会把这 12 个门槛拆开显示，并分别给出英文速度和代码符号的近 7 轮变化、当前卡点和下一轮建议。

`Today` 的 `今日执行协议` 会把策略转成可执行规则：当前主线练什么，满足哪些条件才能升级，单轮准确率或退格失控时如何降级保护，以及每周需要复盘哪些数据。这样可以避免在底座未达标时过早追中文五笔速度，也避免底座达标后继续停留在英文/代码舒适区。

`开始今日训练` 主按钮会把当前执行协议转成今日队列：英文门槛未过时队列从英文开始，英文已过但代码未过时从代码开始；英文/代码底座全部达标后，队列才把五笔或真实中文放到主位。

侧边栏 `Practice` 入口也服从当前执行协议。没有正在进行的队列或明确选中的材料时，它会进入当前主线模式，而不是沿用一个可能过时的手动模式；手动模式按钮仍可用于临时切换。

自由练习中如果手动切到非当前主线模式，Practice 会显示协议提示：底座未达标时，五笔和真实中文只能低量维护，不能替代英文/代码 80 CPM 主训练；五笔主线解锁后，英文/代码手动练习也会标为维护或临时加练。

如果当前训练已经开始并且输入框里有内容，`Esc` 退出、`n` 下一组、`r` 重练、手动切换模式或直接关闭页面都会先确认。已经保存完成的轮次不再拦截；保存中的轮次会等待保存结束，避免半写入状态被误操作打断。

`Practice` 完成一轮后会在本次诊断里生成 `训练处方` 和 `下一组建议`。训练处方会先判断准确率、提示依赖、退格、长停顿和提速空间，并给出下一轮保护规则；下一组建议优先服从当前执行协议：例如英文门槛刚过但代码仍缺样本时，下一组会转向代码；主线模式没有材料时才回退到当前模式。这样训练结束页可以直接进入下一组，不需要用户自己重新判断。

长停顿不仅统计两次输入之间的卡顿，也会统计最后一次输入到点击完成之间的停顿。如果输入完后停住思考再完成，本轮诊断仍会把这段停顿计入 `长停顿` 和 `maxPauseMs`，避免漏掉材料尾部卡点。

英文/代码训练中，训练台会实时显示 `底座有效性`：训练时长是否达到 12 秒、材料和实际输入是否都达到 10 个输入单位、是否使用提示、是否发生粘贴或大段突增输入。状态为 `可计入底座` 时，完成后才会进入 5 轮基线、20 轮稳定窗口和 14 天复测统计；状态为 `还不能计入底座` 时继续输入或等到有效时长；状态为 `本轮不计入底座` 时仍可保存训练记录，但只作为热身或异常输入记录。

训练处方现在会明确给出单轮决策：`通过，可继续`、`热身通过`、`重练当前组`、`无提示重练`、`降速保护`、`停顿复盘` 或 `提速重练`。只有真正通过的轮次会默认推进今日队列；英文/代码短轮次、发生粘贴或大段突增输入的轮次会显示为热身通过，可以继续队列但不计入底座基线、稳定窗口或复测有效期。如果准确率、提示、退格、长停顿或英文/代码 80 CPM 底座不达标，训练台会把默认动作改成重练当前组，并在侧栏保留“仍然继续”的弱入口，方便临时跳过但不误导日常训练。

完成英文或代码训练后，本次诊断会单独显示“计入底座”或“不计入底座”。计入底座只说明这轮满足样本质量要求，会进入 5 轮基线、20 轮稳定窗口和 14 天复测统计；速度、准确率和退格仍会按实际表现参与门槛判断。使用提示、粘贴、自动填充、短轮次或过短材料都会显示为不计入底座。这样可以区分“这轮有效但需要重练”和“这轮只是热身，不推进底座”。

粘贴事件和大段突增输入会被记录为异常输入，但同一次显式粘贴不会再被随后的输入变化重复计算为第二次突增输入；如果没有浏览器粘贴事件，系统仍会用突增输入规则捕获自动填充或异常大段输入。

`Analytics` 的 `周复盘` tab 会按最近 7 天自动生成复盘：训练天数、分钟数、速度变化、模式投入、弱项、主要风险、下周训练计划和验收标准。下周计划按每日 20 分钟、每周 5 天生成，直接列出每个训练块的分钟数、周投入和通过标准；点击计划块里的 `开始此块` 会按分钟数估算生成多组材料，例如 8 分钟英文块默认生成 4 组材料，材料不足时只使用已有的不重复材料。点击 `导出周复盘` 会下载 Markdown 文件，便于保存到日志、项目笔记或同步目录。它的核心判断仍然服从底座策略：英文/代码门槛未过时继续冲底座；门槛通过后，下周主线转为五笔编码、弱字弱词和真实中文，同时保留较弱英文或代码的 100 CPM 维护块。

周复盘里的速度卡片和风险记录以当前执行协议的主线模式为准：底座未达标时看英文或代码，底座达标后看五笔或真实中文。混合模式总体均值仍会保留在复盘数据里，但不会单独作为升级依据，避免用英文速度掩盖代码短板，或用五笔低速样本拉低英文/代码底座判断。

- 第 0 阶段：安装验收，只确认依赖、启动器、材料和同步出口，不作为训练主线状态
- 第 1 阶段：英文/代码速度底座
- 第 2 阶段：五笔编码准确
- 第 3 阶段：真实中文迁移
- 第 4 阶段：生产切换

`Today` 的“下一组建议”现在先服从执行协议，再调用自适应队列挑材料。英文/代码底座未过时，推荐顺序会固定为主线英文或代码、另一项底座补足、Vim/命令维护、五笔低量维护；即使某个五笔弱项得分很高，也不会抢到主训练位。底座通过后，推荐顺序再切到五笔编码、真实中文、英文/代码维护。每个模式内部仍会综合同一材料或同一目标文本的历史记录：准确率不足、使用提示、退格偏高、长停顿、近期重复错误会提高优先级；刚刚稳定练过的项目会降低优先级；间隔较久的项目会进入复习。

点击 `开始今日训练` 会把当前剩余的每日计划转成 `今日队列`，Practice 顶部显示当前进度和计划步骤，例如 `今日队列 1/4 · 英文速度底座 8m`。自适应推荐只负责在每个计划步骤里挑具体材料，队列顺序仍以今日计划为准。完成一组后诊断区会优先显示队列下一组，按 `n` 或点击 `开始下一组` 会按顺序推进；如果从推荐列表里的某一条开始，队列会从该条所属模式开始旋转计划。队列结束后回到 Today，此时可以直接关闭应用，或按当前执行协议自由加练。

Today 队列和周计划队列都会按计划分钟估算应练组数，例如英文 8 分钟默认需要 4 组。Today 会在开始前显示队列可执行性：今日计划需要几组、当前材料可覆盖几组、每个模式缺几组，并把英文/代码底座材料是否满足 80 CPM 解锁线一起显示。Practice 的计划标签和页面通知仍会显示缺口。本次训练会先使用已有材料继续推进，但应到 `Materials` 补充对应模式，避免 20 分钟计划实际只练了几分钟却被误当作完整执行。

训练台快捷键：

- `Enter`：完成当前题；完成后按处方进入下一组或重练
- 完全匹配目标：自动保存当前题
- `Ctrl/⌘ + Enter`：完成代码训练
- `Esc`：退出到 Today
- `r`：重练
- `n`：下一组
- `s`：打开统计
- `?` / `z`：五笔提示

## 推荐训练节奏

默认每日 20 分钟。如果英文/代码尚未达标：

- 英文速度底座：8 分钟
- 代码符号：7 分钟
- Vim/命令：3 分钟
- 五笔低量维护：2 分钟

英文和代码都稳定到 80 CPM 以上后，切换为五笔主线：

- 字根/简码：4 分钟
- 弱字弱词：5 分钟
- 真实中文：6 分钟
- 专业词组：3 分钟
- 英文/代码维护：2 分钟

可选每日 30 分钟时，按同一比例增加英文/代码或五笔主线材料。

## 设置与数据维护

`Settings` 可以修改：

- 设备名：用于区分家里 Mac、公司 Ubuntu 等设备
- 每日目标分钟：限制在 5 到 180 分钟
- 同步目录提示：用于记录你计划放同步包或后续自动同步目录的位置

导入同步包或 `typinglab-sync-folder-*.json` 时，Settings 会先显示待确认预览：来源设备、导出时间、文件内数据量、确认后本机总量、新增/跳过的 sessions、events 和 materials。预览还会显示确认后的数据健康状态、同步建议、设备数和潜在问题。只有点击 `确认合并` 后才会写入本机 IndexedDB；取消导入不会修改本机数据。重复导入会跳过已有 `eventId` / `sessionId`，同名同内容材料会跳过，同 ID 但内容不同的材料会生成新 ID。

`写入同步目录` 和 `读取同步目录` 不会启动常驻后台同步进程。Web MVP 在本次打开期间可以复用已授权目录做自动写入；关闭页面或浏览器回收权限后，需要重新点击 `写入同步目录` 授权。跨电脑同步仍交给 Syncthing、iCloud、Dropbox、公司网盘或手动文件传输。

材料删除通过 `material_deleted` 事件同步。合并时只有未被训练记录引用、且删除事件不早于材料更新时间的材料才会移除；已被历史训练引用的材料会保留，避免旧训练记录出现悬空材料引用。

如果 `待导出会话`、`待导出事件` 或 `待导出材料` 不为 0，说明本机 IndexedDB 里有尚未进入同步包/同步目录的事实流。跨设备训练时，完成一段训练或整理材料后优先使用 `写入同步目录`；浏览器不支持时再用 `导出同步包` 或 `同步目录清单`。当存在多设备数据且超过 7 天没有从另一台设备导入时，健康面板会把下一步动作标为 `导入另一台设备`。

Today、连续天数、趋势图、周复盘和同步目录 session 文件名都按本机本地日历日归属训练记录，不按 UTC 日期切分。这样晚上训练或跨 Mac/Ubuntu 合并后，今日剩余分钟和周复盘不会因为 ISO 时间跨日而错位。

新的训练记录会在输入间隔超过 1500ms 时追加 `long_pause` 事件，payload 里包含停顿毫秒数和发生位置。旧训练记录仍可用于速度、准确率、退格和停顿数量统计，只是无法回推具体停顿位置。

清空本机训练记录需要输入 `清空` 才能启用按钮。该操作只删除本机 sessions/events，保留材料和设置；清空前建议先导出同步包。

## 验证

推荐在 Mac/Ubuntu 新 clone、更新代码或准备发布前先跑本地自检：

```bash
npm run doctor
```

`npm run doctor` 会检查 Node.js/npm、依赖安装和本地启动/构建/测试工具、`.node-version`、GitHub Actions 完整自检、关键 npm scripts、`package.json` engines、训练数据导出忽略规则、本地同步产物残留、启动脚本语法、关键 smoke 流程和生产构建。缺依赖时默认自动执行 `npm ci`；公司网络受限时可以用 `TYPINGLAB_AUTO_INSTALL=0 npm run doctor` 关闭自动安装。发布前想加全量测试，可以运行：

```bash
npm run doctor:full
```

GitHub Actions 会在 Ubuntu 上执行同一套完整自检：`npm ci` 后运行 `npm run doctor:full`。本地默认 Node 版本写在 `.node-version`，当前为 Node.js 20；`package.json` 声明 Node.js >=18、npm >=9，`.npmrc` 开启 `engine-strict=true`，启动器也按 Node.js 18+ 校验。

也可以单独运行：

```bash
npm run test:smoke
npm test
npm run build
```

`npm run test:smoke` 只跑关键产品路径：内置材料库覆盖度，英文/代码底座门槛和训练中实时有效性判断、Today 执行协议队列、队列可执行性和材料缺口汇总，Practice 手动模式偏离主线提示、未完成输入保护、实时 CPM/准确率/进度和自动完成判定，周复盘生成下周计划、计划块按分钟量生成多组训练队列、周复盘 Markdown 导出，低质量训练触发处方保护且阻止误推进，文本材料和 Rime 五笔词库导入校验，专业材料生成器拆分中文、英文、代码和 Vim/命令候选，同步包校验、材料版本冲突预览、同步健康对会话/事件/材料待导出的判断，同步目录导出、清单读回、manifest 最后写入、manifest 完整性校验、半同步拒绝，Mac/Ubuntu 各训练一组后的双向合并指纹一致和重复导入去重，导出文件名的时间戳/摘要对账，以及 service worker 对旧构建 assets 的缓存清理。

当前构建把代码训练用的 Monaco Editor 做成按需加载；普通五笔/中文训练不需要先加载完整代码编辑器。

生产构建会复制 `public/manifest.webmanifest` 和 `public/sw.js`，浏览器支持时可安装为 PWA。Service worker 对页面 HTML 使用网络优先，避免产品更新后旧缓存继续引用已删除的构建资源；静态资源仍会缓存。每次成功拿到最新 HTML 后，service worker 会解析当前 `/assets/` 引用并清理缓存里不再被 index 引用的旧构建资源，减少 Mac/Ubuntu pull 更新后的离线缓存残留。开发模式不注册 service worker，避免调试时缓存旧代码。
